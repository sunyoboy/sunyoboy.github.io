## 为什么是系统不是方法

你知道该质疑——但在真实会议里，早上九点的质疑质量和下午四点的质疑质量，不应该差一倍。一个系统让你疲惫时也能执行。

```
团队里的雷 = 别人的决策 + 你的沉默
你自己没犯任何错，项目炸了你一起扛。

不系统的质疑方式：靠感觉、靠心情、靠勇气
系统的质疑方式：每一层都有检查点。过一遍，红灯亮了就出声。
```

## 七层过滤系统

```
第一层  事实基础       我们说的这件事，有哪些是事实，哪些是推测？
第二层  关键假设       如果有一个假设是错的，哪个最致命？
第三层  不可逆风险     决定了就不能回头的，是什么？
第四层  逆向验证       如果这个方案三个月后失败了，最可能的原因是什么？
第五层  沉默信号       有谁还没出声？为什么？
第六层  最小试错       能不能用更小的代价验证同样的假设？
第七层  决策边界       这件事我们最多能承受多少损失？
```

---

### 第一层：事实基础

| 检查点 | 为什么重要 |
|------|------|
| 把会上说的每一句话，分成"已验证的事实"和"我们的推测"两列 | 90% 的团队雷来自把推测当成事实来用 |
| 如果有人说了"很明显"、"肯定是"、"大家都不" —— 这三个是事实还是推测？ | 这三个短语是推测的黄金话术。每出现一次，就应该被标记一次 |

> 不需要在会上当众指出来。自己在心里过一遍 —— 你已经知道这间屋子里有多少是事实、有多少是猜测。

---

### 第二层：关键假设

每一个方案都建立在一些没被验证的假设上。不找出来，方案就是沙做的城堡。

```
一个完整的方案通常有三个假设层级：

表面假设： "用户会喜欢这个功能"
中层假设： "我们有能力把它做出来"
底层假设： "做出来之后真的能解决用户的问题"

三个都错了 = 全部白做。
只有底层假设对了 = 方向对了，但要调整做法。
只有表面假设错了 = 需要重新理解用户，但方向和方法都在。
```

**工具**：对一个方案，问三次"如果这个假设不成立，什么会先坏掉"。第一次问的是风险，第三次问的是根本错误。

---

### 第三层：不可逆风险

```
可逆的风险 = 错了能改 = 可以容忍
不可逆的风险 = 错了就完了 = 必须出声

例：
  下周一上线一个功能 → 发现 bug → 回滚 → 可逆
  签了三年独家合作 → 发现选错了 → 赔违约金 → 不可逆
  把数据库删了 → 没备份 → 不可逆
  把核心架构推倒重来 → 所有人等三个月 → 不可逆
```

> 只有不可逆的风险，才值得你打破团队沉默。可逆的可以事后复盘。

---

### 第四层：逆向验证

不预测未来，模拟未来。

```
"假设现在是三个月后。这个项目已经完全失败。
 现在我们坐在这间屋子里复盘：
 是什么导致了这个结果？"

这个问法有两个好处：
 ① 从"证明你错了"变成"一起想象一个坏未来"
 ② 所有隐藏的担心在这一刻都有了出口
```

**这是你在团队里能用出来的最有效的质疑工具——因为它不攻击任何人，它攻击一个虚构的未来场景。**

---

### 第五层：沉默信号

| 沉默类型 | 含义 | 应对 |
|------|------|------|
| 最资深的人没出声 | 可能在观察，可能有不同意见但不方便说 | 会后找他聊 |
| 最了解细节的人没出声 | 他看到的坑比任何人都多 | 必须单独问 |
| 所有人都沉默了一下 | 有人压住了群体异议 | 刚才那段话里有雷 |
| 你自己沉默了 | 有什么在阻止你？怕说错？怕孤立？找出原因 | 会后找一个人先对一下 |

> 一个房间里最危险的噪音是沉默。会后的咖啡机旁边说的话，往往比会上的 PPT 更接近真相。

---

### 第六层：最小试错

交易上有"500 份试仓 vs 全部满仓"。团队项目也一样：

```
全量方案：            最小试错方案：
"我们把整个模块重写"   "先重写最核心的 10%，跑通全流程"
"三个月后完成"        "两周后出一个只测一件事的原型"
"面向所有用户"        "找 5 个用户试，不公开"
```

> 如果团队不愿意做最小试错，说明团队本身对这个方案的方向就没有信心 —— 在逼自己"一次性赌对"。这本身就是最大的雷。

---

### 第七层：决策边界

交易上有止损（每笔最多亏 1%）。团队决策也需要止损：

```
提案的时候同时明确：
  "这个项目一旦触发什么条件，我们就停下来？"
  "上线后多少天没有正面反馈，我们不再追加资源？"
  "预算超了多少，必须重新申请而不是自动追加？"

没有边界的项目 = 永远不会停止的投资 = 迟早变成沉没成本陷阱
```

## 七层检查清单

一次会议过一遍，不需要逐条念，两分钟就够了：

```
□ 事实和推测分开了吗？
□ 最致命的一个假设是什么？如果它错了会怎样？
□ 有不可逆的决策吗？是的话我出声了吗？
□ 如果三个月后失败了，第一原因是什么？
□ 谁还没出声？为什么？
□ 能不能先拿 10% 的成本先试？
□ 止损条件设了吗？触发条件是什么？
```

> 第六感告诉你"这方案不太对"的时候，至少有两层已经亮了红灯。你不需要找到错在哪一层——你只需要知道"红灯亮了，所以我现在需要出声"。

## 四个低摩擦质疑方式

### 1. 问"缺什么"——不问"你错了"

"你是不是搞错了" → 对方防御全开。

换一个问法：

```
"这个方案在什么情况下会出问题？"
"如果有一条信息我们还没拿到，缺的是什么？"
"三个月后回看这个决策，什么事会让我们后悔？"
```

> 效果一样——都是在找风险。但你反对的不是他，你在找一个你们两个都还没看到的盲区。

### 2. 角色切换

你做过代码审查（05-22 的 uni-app review）。带着同样的眼睛看项目计划：最坏能坏到什么程度？最确定的一个假设，如果它错了呢？有没有一个东西大家都在假设但没人验证过？

### 3. 把"我担心"换成"我们试试"

```
"我觉得这个方案风险太大"        →  你是阻力
"能不能用一个更小的场景先试一下"  →  你是推动力

两个表达的是同一件事：我不确定。
第二个让团队不用在"信不信你"和"信不信他"之间选边站。
```

### 4. 找盟友

会后找团队里经验最丰富的那个人问一句：

> "刚才那个方案，你觉得有没有什么我没看到的坑？"

如果他说"有"——你现在有一个强有力的盟友。如果他说"没有"——那你真的可能想多了。

## 什么时候必须出声

| 场景 | 做法 |
|------|------|
| 你看到的风险是**不可逆**的（一旦发生无法回滚）| 必须出声。不说就是你的责任 |
| 你看到的风险是**有人在刻意回避**| 私下先找那个人聊，再决定要不要升级 |
| 你看到的风险是**方向性**的（不是细节问题）| 出声。细节可以事后改，方向错了整船人一起沉 |
| 你看到的风险是**你自己也没把握的**| 用问句。不要用陈述句。"有没有可能……" |

## 与交易系统的精确映射

```
交易七层                    质疑七层
──────────                ──────────
第零层 仓位合规             第三层 不可逆风险（能不能承受这笔损失）
第一层 成交额情绪           第一层 事实基础（这是事实还是情绪）
第二层 MA5偏离度            第二层 关键假设（趋势方向对吗）
第三层 相关性               第四层 逆向验证（反过来想：如果趋势反了）
第四层 事件日历             第五层 沉默信号（关注"没发生的"和"没说的"）
第五层 全球比价             第六层 最小试错（先下小仓位试）
第六层 月度效应             第七层 决策边界（最大回撤、止损）
第七层 执行指令             第七层 决策边界（退出条件）
```

> 同一套脑回路。你只是在用"质疑自己的交易决策"的方式质疑一个团队决策。区别只在于说出来的方式——对自己可以说"这个仓位逻辑有漏洞"，对团队说"我们试试能不能先用 10% 的成本验证一下"。
